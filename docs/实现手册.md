# 实现手册

## 1. 目标

这个项目实现的是一个项目级 agent 中枢，而不是单纯的聊天机器人。

目标是：

- 飞书和 Telegram 作为统一入口
- 入口面对的是项目，不是单个 provider 会话
- 项目内部由 Claude 和 Codex 协作
- 最终对用户的汇报由 Claude 统一输出
- 底层长期会话由 `ccb/askd` 承载

## 2. 当前运行结构

```text
Remote Chat
  Telegram / Lark
        |
        v
CCConnectAdapter
  project worker
        |
        +--> Claude lane
        |     discussion / validation / reporting
        |
        +--> Codex lane
              implementation / execution
        |
        v
LocalExecutorSessionPool
        |
        v
Executor
  ccb / askd / direct fallback
```

## 3. 核心对象

### 3.1 workspace

项目级上下文对象，保存：

- `workspace_id`
- `path`
- `backend`
- `transport`

### 3.2 workspace_session

某个 chat 在某个 workspace 下的运行态，保存：

- `active_task_id`
- `executor_session_id`
- `claude_session_id`
- `codex_session_id`
- `phase`
- `conversation_summary`

### 3.3 phase

当前已落地的状态机：

- `discussion`
- `ready_for_execution`
- `executing`
- `reviewing`
- `reported`

### 3.4 agent_messages

Claude/Codex 各自的最近消息流。

作用：

- Claude 只看到 Claude 自己的讨论/汇报上下文
- Codex 只看到 Codex 自己的执行上下文

### 3.5 task_handoffs

结构化交接对象表。

当前类型：

- `discussion_brief`
- `execution_packet`
- `review_verdict`

## 4. 消息流

### 4.1 `/write`

1. 创建 root task
2. phase 设为 `discussion`
3. 把用户消息写入普通消息流
4. 把用户消息写入 Claude agent 流
5. 使用 Claude lane 执行

### 4.2 普通文本续聊

1. 如果当前 workspace 有 active task
2. 默认继续当前 task
3. 默认走 Claude lane
4. 使用 Claude 专属最近记忆

### 4.3 `/execute`

1. 从 Claude 讨论流提取 `discussion_brief`
2. 生成结构化 `execution_packet`
3. phase 设为 `executing`
4. 把执行包交给 Codex lane
5. Codex 返回结果
6. phase 进入 `reviewing`
7. Claude 基于：
   - `discussion_brief`
   - `execution_packet`
   - `codex_output`
   做最终 review
8. 生成 `review_verdict`
9. phase 落到 `reported`

## 5. ccb 的位置

`ccb` 不负责最上层调度，`project worker` 才负责调度。

`ccb` 在这套里的位置是：

- 承载 Claude/Codex 的长期后台会话
- 按 `work_dir + provider + project_id` 路由到正确 pane/session
- 让 `claude_session_id` / `codex_session_id` 不只是逻辑 ID，而能映射到底层真实会话

### 5.1 现在已经接入的部分

当前 `worker_session -> AskExecutor` 会注入：

- `CCB_SESSION_ID`
- `CCB_WORK_DIR`
- `CCB_RUN_DIR`

再结合 `askd_client` 自己的：

- `work_dir -> ccb_project_id`
- `provider -> pane registry`

就能把项目 worker 的双 agent session 对到 CCB 的实际会话。

### 5.2 为什么这样接

因为你要的是：

- 项目级调度
- agent 级长期会话
- Claude/Codex 平级协作
- 外部统一项目视图

如果让 `ccb` 直接当最上层调度器，就很难维护：

- phase
- handoff
- chat/workspace 绑定
- 项目级 summary

所以正确分层是：

- `worker` 管流程
- `ccb` 管底层会话

## 7. 当前限制

虽然已经接上了 `ccb` 环境，但当前执行层仍然是：

- `AskExecutor` 前台调用 `ask --foreground`
- 或 direct fallback

所以它已经进入 `ccb` 路由层，但还没有做：

- 直接读取 mounted 状态
- 显式检查 pane 存活
- 强制只走 mounted session 不 fallback

## 8. 下一步建议

最值得继续做的三件事：

1. 增加 `ccb mounted` 健康检查
2. 为 `transport=ccb` 提供更严格的行为：
   - 优先只走 CCB
   - 未挂载时明确报错
3. 增加 `/worker` 或 `/sessions` 命令，直接查看：
   - current phase
   - claude session
   - codex session
   - handoff count
   - mounted status

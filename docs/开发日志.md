### local-native 退出阶段数据库打开失败修复

- 问题现象：
  - `local-native --provider codex` 退出后在 postrun 同步阶段报错：
  - `sqlite3.OperationalError: unable to open database file`
- 根因：
  - runtime 数据库路径默认是相对路径 `var/db/agent-swarm-hub.sqlite3`。
  - `local-native` 运行中切换了 `cwd`（`os.chdir(work_dir)`），导致后续 `SessionStore` 在错误目录下解析数据库路径。
- 修复：
  - `SessionStore` 初始化时将数据库路径统一解析为绝对路径（`expanduser().resolve()`）。
  - 移除 `local-native` 中不必要的 `os.chdir(work_dir)`，由 `subprocess.run(..., cwd=work_dir)` 控制 provider 工作目录即可。

### 回归测试补齐（避免同类问题再次出现）

### 依赖噪音收口（Lark 按需加载）

- 现状确认：
  - `lark-oapi` 当前已是最新 `1.5.3`，该版本内部仍使用 `pkg_resources.declare_namespace`。
  - 因此在任何命令启动时若提前 import Lark 组件，会出现 `pkg_resources is deprecated` warning。
- 处理方式：
  - 将 CLI 中 `LarkWebSocketRunner` 改为按需 lazy import，仅在执行 `lark-ws` 时加载。
  - 将 `agent_swarm_hub.__init__` 中 `LarkService/LarkWebSocketRunner` 改为 `__getattr__` 懒加载。
- 结果：
  - `project-sessions cleanup-runtime` 等非 Lark 命令不再触发该 warning。
  - Lark 相关命令和测试功能保持不变。

### swarm 主线路径改成 trigger -> Claude orchestrator -> Codex workers

- 本地与远程的 swarm 语义进一步收口，不再把“当前终端 = 主控”作为默认模型。
- 当前终端现在明确只是 `trigger`：
  - 负责进入项目
  - 负责发起讨论
  - 负责触发 swarm
- 一旦任务进入协作执行，系统的正式主控改成固定 `Claude orchestrator`：
  - `planner = claude`
  - `orchestrator = claude`
  - `executor = codex`
  - `reviewer = claude`
- 这意味着正式结果的默认归档与回报目标也统一成 `claude`，不再跟随当前 trigger 终端漂移。

  - `Trigger Terminal`

  - `Review Return Target`

### tmux orchestrator / worker 启动链补上

- 新增 `swarm_launch.py`
  - 负责确保项目下存在指定 provider 的 tmux pane
  - 优先复用已存在 pane
  - 缺失时自动通过 `scripts/start-chat.sh` 拉起新的 tmux session/window
- 进入复杂任务执行时：
  - 先确保有 `Claude orchestrator` pane
  - 再进入 planning
- 进入 sub-agent 执行时：
  - 每个 `Codex` worker 在执行前也会先确保有自己的 tmux pane
  - `worker_launch` 状态会被写进 `subagent_packet` / `subagent_result`
- 这意味着当前链路已经从“只有角色语义”升级为“真实 tmux 启动 + 结构化状态记录”

### tmux worker 清理与 dashboard 分层展示

- `worker` 的 tmux 启动结果现在会返回完整句柄：
  - `launch_kind`
  - `session_name`
  - `window_index`
  - `pane_id`

  - 新起的 worker session/window 会被清理
  - 已存在的 pane 不会被误杀
- `subagent_result` 会额外记录：
  - `worker_cleanup.status`
  - `worker_cleanup.target`
- dashboard 的 `Swarm` 区块进一步拆层：
  - `Orchestrator Launch`
  - `Workers`
  - `Verification`
- 现在可以更清楚地判断：
  - 子任务是否真的被唤起
  - tmux pane/session 是否被正确清理

### 死 codex pane 复用降噪与真实跑测结论

- 对 `codex + auto/askd` 增加了短 TTL 的死 pane 结果缓存：
  - 同一项目在短时间内如果已经确认旧 pane 已死
  - 后续同一轮 swarm 不再反复探测同一个坏 pane

- `subagent_result` 现在会稳定带上：

  - `worker_cleanup`
- 真实跑测已经确认：
  - `claude` discussion / planning 正常写入 runtime DB
  - `project_memory` 会以 compact snapshot 形式分给子 agent
  - `codex` worker 会被真实唤起到 tmux
  - 任务结束后 worker session 会被真实清理
  - 若 `codex exec` 超时，也仍会写回 `subagent_result + worker_cleanup`
